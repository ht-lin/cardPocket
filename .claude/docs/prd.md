# CardPocket — 产品需求文档（PRD）

> 基于 SPEC.md v1.2 | 面向开发者，描述"做什么"而非"怎么做"

---

## 产品定位

CardPocket 是一个**公开社区应用**，允许任意用户注册，将会员卡、门票、优惠券等实体卡片的条码数字化保存，并通过好友系统与家人/朋友共享卡片访问权限。非商业化，目标用户以欧洲用户为主（GDPR 合规）。

---

## 用户角色

| 角色 | 描述 |
|------|------|
| 未验证用户 | 已注册但邮箱未验证，可登录，不可创建卡片或添加好友 |
| 已验证用户 | 邮箱已验证，可使用全部功能 |
| Owner | 卡片创建者，拥有卡片的完整控制权 |
| Viewer | 被 Owner 共享了某张卡片，只能查看和设置私有昵称 |

---

## 核心用户故事

### 卡片管理

- **US-01**：作为用户，我可以通过手动输入条码创建卡片，以便数字化保存实体卡片
- **US-02**：作为移动端用户，我可以用手机相机扫描条码快速添加卡片
- **US-03**：作为用户，我可以修改卡片名称
- **US-04**：作为用户，我可以删除不再需要的卡片
- **US-05**：作为用户，我可以在无网络时查看并展示卡片条码（离线）
- **US-06**：作为用户，我可以搜索自己的卡片（Phase 2）
- **US-07**：作为用户，我可以自定义卡片的颜色/图标外观（Phase 2）
- **US-08**：作为用户，我可以设置卡片有效期，过期后自动归档（Phase 2）

### 好友与共享

- **US-09**：作为用户，我可以通过用户名或邮箱精确搜索其他用户并发送好友请求
- **US-10**：作为用户，我可以接受或拒绝收到的好友请求
- **US-11**：作为用户，我可以解除好友关系（同时撤销所有相关共享）
- **US-12**：作为卡片 Owner，我可以将卡片共享给好友，让对方也能展示条码
- **US-13**：作为卡片 Owner，我可以查看所有共享该卡的成员，并可随时移除
- **US-14**：作为 Viewer，我可以查看并展示被共享给我的卡片条码
- **US-15**：作为 Viewer，我可以为共享卡片设置私有昵称（只有自己可见）
- **US-16**：作为 Viewer，我可以主动退出某个卡片的共享

### 账户与安全

- **US-17**：作为用户，我可以通过邮箱+密码注册账户
- **US-18**：作为用户，注册后我必须验证邮箱才能完整使用应用
- **US-19**：作为用户，我可以修改用户名和密码
- **US-20**：作为用户，我可以删除我的账户并清除所有数据（GDPR）
- **US-21**：作为移动端用户，离开 App 一段时间后需要生物识别重新解锁（可配置）

### 图片录入（Phase 3）

- **US-22**：作为用户，我可以从相册选取含 QR Code 的图片，自动解析条码
- **US-23**：作为用户，我可以从其他 App（如相机/微信）通过系统分享功能直接把图片发送给 CardPocket 解析

## 功能需求

### FR-01：邮箱验证门控

未验证用户调用以下端点时返回 403：
- POST /api/cards（创建卡片）
- POST /api/friendships（发送好友请求）

其余端点不受影响（可查看列表、管理已有数据等）。

### FR-02：好友前置条件

CardShare 创建时，后端必须验证 `card.owner` 与 `viewer` 之间存在 `Friendship(status=ACCEPTED)`，否则返回 403。

### FR-03：好友解除级联

DELETE /api/friendships/{id} 成功后，数据库层面自动（CASCADE）删除该好友对之间所有 CardShare 记录。

### FR-04：条码不可修改

Card 创建后，`barcodeType` 和 `barcodeContent` 字段在任何 PATCH 请求中均被忽略（不报错，静默忽略）。

### FR-05：增量同步

GET /api/cards?updatedAfter={timestamp} 必须同时返回：
- `updated`：时间戳后新增或修改的卡片（含对 Viewer 新增的共享卡片）
- `deleted`：时间戳后被删除的 Card UUID 或对当前用户撤销的 CardShare 对应 Card UUID

### FR-06：Viewer 昵称隔离

`CardShare.viewerNickname` 只有对应 Viewer 自己可读写，Owner 获取卡片详情时不会看到 Viewer 的昵称数据。

### FR-07：用户搜索结果

GET /api/users/search 永远只返回 `id + userName`，即使搜索词是 email，响应中也不包含 email 字段。无匹配时返回空数组（不返回 404）。

### FR-08：卡片数量上限

每用户 Card 数量（Owner）上限 200 张。超出时 POST /api/cards 返回 422，错误信息说明上限。

## 非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | 主要 API（GET /api/cards）P95 < 200ms（VPS 正常负载下） |
| 离线 | 原生 App 无网络时必须能展示所有本地缓存的卡片条码 |
| 安全 | 所有敏感数据走 HTTPS；原生本地缓存存 expo-secure-store；JWT Access Token 不持久化 |
| GDPR | 欧盟用户数据存储在欧盟节点；删除账户 24 小时内完成数据清除 |
| 可用性 | 单 VPS 部署；无 SLA 承诺（个人项目），但应有基础监控和数据库备份 |

## 约束与排除项

- **不做标签系统**：用搜索代替分类
- **不做卡片备注**：卡片名称本身已具描述性
- **不做 Apple/Google Wallet**：依赖商家支持，不可控（Phase 4 可探索）
- **不做实时同步**：采用"进入前台时拉取"的简单策略，不做 WebSocket/SSE
- **不做 OAuth**：仅 email + password，降低初期复杂度
- **Web 版不是 MVP**：Phase 3 才开始
